## 2. 功能需求

### 2.1 必须功能

| 功能 | 说明 |
|------|------|
| 站点列表 | 展示当前面板已管理的站点记录，包括名称、域名、类型、状态、根目录或代理目标。 |
| 新增站点 | 支持新增站点，至少支持两种模式：**静态站点**、**反向代理**。 |
| 编辑站点 | 支持修改已有站点的域名、根目录、代理目标、监听端口、启用状态等基础配置。 |
| 删除站点 | 支持删除站点记录；如已生成配置，应同步删除或禁用对应 Nginx 配置。 |
| 配置生效 | 对站点配置生成 Nginx 配置文件，执行校验并重载 Nginx。 |
| 状态反馈 | 当配置校验失败、文件写入失败或重载失败时，前端应给出明确错误信息。 |

### 2.2 站点类型

#### 静态站点

用户需要提供：

- 站点名称
- 域名
- 根目录
- 监听端口（默认 80）
- 是否启用

适用场景：

- 前端静态页面
- 简单 HTML 资源托管

#### 反向代理站点

用户需要提供：

- 站点名称
- 域名
- 代理目标地址，例如 `http://127.0.0.1:3000`
- 监听端口（默认 80）
- 是否启用

适用场景：

- 代理 Node / Python / Java 等后端应用
- 将公网访问入口统一交给 Nginx

### 2.3 交互与体验

- 列表页需支持快速查看站点基本信息。
- 新增/编辑应使用统一表单，避免重复维护。
- 删除操作需二次确认，并提示这是系统级配置变更。
- “保存”与“保存并生效”可以分离，也可先采用“保存即尝试生效”的简单方案；但必须明确展示结果。
- 若 Nginx 未安装、配置目录不可写、校验失败等，必须提示具体原因。

### 2.4 可选 / 后续

- HTTPS 与证书管理
- 多域名绑定
- 自定义 Nginx 片段
- 访问日志与错误日志查看
- 站点启停细粒度控制

## 3. 接口约定

### 3.1 站点列表

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/sites | 需登录；返回当前所有站点记录。 |

**响应示例：**

```json
{
  "items": [
    {
      "id": 1,
      "name": "blog",
      "domain": "blog.example.com",
      "site_type": "proxy",
      "root_path": null,
      "proxy_target": "http://127.0.0.1:3000",
      "listen_port": 80,
      "enabled": true,
      "status": "active",
      "updated_at": "2026-03-07T12:00:00"
    }
  ]
}
```

### 3.2 获取单个站点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/sites/{id} | 需登录；返回单个站点的详细配置。 |

### 3.3 新增站点

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/sites | 需登录；创建站点记录，并尝试生成/更新 Nginx 配置。 |

**请求示例：**

```json
{
  "name": "blog",
  "domain": "blog.example.com",
  "site_type": "proxy",
  "root_path": "",
  "proxy_target": "http://127.0.0.1:3000",
  "listen_port": 80,
  "enabled": true
}
```

### 3.4 编辑站点

| 方法 | 路径 | 说明 |
|------|------|------|
| PUT | /api/sites/{id} | 需登录；更新站点记录，并重新生成/更新 Nginx 配置。 |

### 3.5 删除站点

| 方法 | 路径 | 说明 |
|------|------|------|
| DELETE | /api/sites/{id} | 需登录；删除站点记录，并删除或禁用对应 Nginx 配置。 |

### 3.6 手动应用配置

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/sites/{id}/apply | 需登录；重新生成当前站点配置并执行校验与重载。 |

### 3.7 统一错误约定

- `400`：请求参数非法，如域名为空、代理地址格式错误、根目录为空
- `401`：未登录或 token 无效
- `404`：站点不存在
- `409`：域名冲突、站点名称冲突等
- `500`：服务端内部错误
- `503`：Nginx 不可用、配置目录不可写、重载失败

**错误响应示例：**

```json
{
  "detail": "Nginx 配置校验失败：server_name 重复"
}
```

## 4. 数据与存储

建议新增站点配置表，例如 `site_configs`。

### 4.1 建议字段

| 字段 | 说明 |
|------|------|
| id | 主键 |
| name | 站点名称，面板内部标识 |
| domain | 域名 |
| site_type | `static` 或 `proxy` |
| root_path | 静态站点根目录；静态站点必填 |
| proxy_target | 反向代理目标地址；代理站点必填 |
| listen_port | 监听端口，默认 80 |
| enabled | 是否启用 |
| config_filename | 生成的 Nginx 配置文件名 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 4.2 数据约束建议

- `name` 建议唯一，避免面板内部混乱
- `domain` 建议唯一，避免生成冲突的 `server_name`
- `site_type=static` 时，`root_path` 必填，`proxy_target` 为空
- `site_type=proxy` 时，`proxy_target` 必填，`root_path` 为空或忽略

### 4.3 系统落地与数据库记录的关系

这里要明确区分两类状态：

- **数据库记录**：面板保存的业务配置
- **系统落地状态**：Nginx 配置文件是否成功生成并生效

建议后续为此保留一个轻量状态字段，例如：

- `draft`
- `active`
- `error`

用于在前端列表页中提示“已保存但未生效”“生效失败”等状态。

## 5. 后端实现要点

### 5.1 路由与模块划分

建议新增：

- `routers/sites.py`：接口定义
- `models/site_config.py`：站点配置模型
- `schemas/sites.py`：请求/响应模型
- `services/nginx_manager.py`：Nginx 配置生成、校验与重载逻辑

这样可以把“业务 CRUD”和“系统命令执行”解耦。

### 5.2 配置生成逻辑

后端不应直接拼接零散字符串到接口函数里，建议集中在模板函数中生成 Nginx 配置。

#### 静态站点示意

```nginx
server {
    listen 80;
    server_name example.com;

    root /var/www/example;
    index index.html index.htm;

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

#### 反向代理示意

```nginx
server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### 5.3 配置写入与生效流程

推荐流程：

1. 校验请求参数
2. 写入或更新数据库记录
3. 生成 Nginx 配置文本
4. 写入指定配置目录，例如 `/etc/nginx/conf.d/`
5. 执行 `nginx -t` 校验配置
6. 若校验通过，执行 `nginx -s reload` 或 systemd reload
7. 若失败，返回明确错误并保留可追踪状态

### 5.4 回滚思路

为避免“保存后直接把系统搞坏”，建议至少考虑以下策略：

- 写文件前保留旧配置内容
- `nginx -t` 失败时恢复旧文件
- reload 失败时返回错误，并标记当前记录状态为 `error`

如果本阶段时间紧，最少也要保证：

- 不通过校验时不执行 reload
- 错误能反馈到前端

### 5.5 参数校验要点

- 域名不能为空
- `site_type` 只能是 `static` 或 `proxy`
- 静态站点要校验根目录
- 代理站点要校验 URL 格式
- 监听端口范围需合法

### 5.6 安全与风险

- 不能允许用户任意写入系统任意路径
- 不能直接信任前端传来的配置片段
- 配置文件名应由后端控制，避免路径注入
- 对系统命令执行结果要做超时与错误捕获

## 6. 前端实现要点

### 6.1 页面结构建议

`/sites` 页面建议至少包含：

- 顶部标题与说明
- 站点列表区
- 新增 / 编辑按钮
- 表单弹层或侧边抽屉
- 删除确认弹窗
- 生效状态提示区

### 6.2 列表页展示信息

建议展示：

- 站点名称
- 域名
- 类型（静态 / 反向代理）
- 根目录或代理目标
- 监听端口
- 状态（已生效 / 未生效 / 错误）
- 最后更新时间

### 6.3 表单交互建议

- 新增和编辑复用同一表单
- 根据 `site_type` 动态切换表单项
- 静态站点显示 `root_path`
- 反向代理显示 `proxy_target`
- 提交后刷新列表，并显示成功或失败提示

### 6.4 状态管理建议

页面至少维护：

- `loading`
- `items`
- `currentItem`
- `form`
- `submitting`
- `error`

如果后续复杂度提升，可再考虑单独抽离组合式逻辑或状态层。

### 6.5 与现有仪表盘风格保持一致

- 延续当前 Dashboard 主题变量与卡片风格
- 保持和 `Monitor.vue`、`Overview.vue` 同级页面一致的间距与层级
- 不在阶段 3 过早引入复杂 UI 库，以免破坏当前风格统一性

- 阶段 4 的数据库管理同样属于系统控制型能力，可复用阶段 3 中的“业务记录 + 系统生效”思路。
