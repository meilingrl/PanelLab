# PanelLab：项目进度评估与迭代计划

本文档基于 2026-03-07 当前仓库代码、配置与文档状态整理，用于回答三个问题：

- 项目现在整体做到了什么程度
- MVP 距离可交付还差哪些关键内容
- 接下来每个周期应该做什么

## 3. 已完成内容

### 3.1 已经可以视为完成

- 用户登录、注册、登出、获取当前用户、修改密码
- JWT 鉴权与前端登录态保存
- 仪表盘整体布局、侧栏导航、主题切换
- 系统监控页面与概览摘要
- 本机 CPU、内存、磁盘、网络采集
- 远程 Linux 监控配置与 SSH 拉取雏形

### 3.2 已有实现但还不能算真正收口

- 远程监控：功能已接通，但缺少真实 Linux 环境的稳定性验证
- 概览页：已有摘要指标，但离真正的“仪表盘总览”还有差距
- 阶段文档：阶段 1、2 较完整，但与代码已有少量不同步

## 4. 当前主要缺口

### 4.1 MVP 缺口

- 网站与反向代理未落地
- 数据库管理未落地

### 4.2 阶段 2 尾项（已收口）

- ~~进程管理未实现~~ → 已实现：本机进程列表（只读，排序/过滤/条数）
- 监控模块 Linux 实机联调为可选验证；概览页接入监控数据为可选

### 4.3 工程化缺口

- 自动化测试基本为空
- 缺少 CI 或统一验收脚本
- 缺少 Docker / 生产部署方案
- 文档与代码存在局部不一致

### 4.4 风险项

- 默认 `secret_key` 与初始化管理员密码偏弱，生产风险高
- 远程 SSH 监控当前更偏实验性实现，安全与兼容性仍需补强
- 前端路由守卫只校验本地 token 是否存在，不校验有效性

## 5. 里程碑重排

建议将后续目标重排为以下 4 个里程碑：

| 里程碑 | 目标 | 完成标志 |
|--------|------|----------|
| M1：阶段 2 收口 | 把监控模块从“能演示”提升到“可稳定验收” | 进程管理已补齐、文档同步、测试清单已补；Linux 联调为可选 |
| M2：网站与反向代理 MVP | 完成站点列表、创建编辑、反向代理配置生成与生效 | 可在 Linux 环境创建站点并完成一次配置应用 |
| M3：数据库管理 MVP | 完成 MySQL 库/用户的基础管理 | 可列表、创建、删除，具备基础校验与反馈 |
| M4：验收与部署 | 补齐测试、部署说明、安全收口 | 可按文档完成部署与一次完整演示 |

## 6. 周期工作内容

默认建议按 **1 周 1 个周期** 推进；若课程时间更紧，可把周期 2 和周期 3 压缩合并。

### 周期 1：阶段 2 收口与验收准备

目标：先把已有功能做扎实，避免带着不稳定监控直接进入阶段 3。

本周期工作：

- 补齐“进程管理”范围确认，至少确定是否进入 MVP
- 完成监控模块的 Linux 实机联调与问题修复
- 修正文档与代码不一致项
- 增加最小验收集：登录、监控接口、远程配置、前端关键页面冒烟
- 收口安全默认值与环境变量说明

本周期产出：

- 阶段 2 完成文档
- 一份可重复执行的测试清单
- 监控模块问题清零或形成明确遗留列表

### 周期 2：网站与反向代理设计 + 后端底座

目标：先搭完整模型和接口，不急于一次做完所有页面细节。

本周期工作：

- 新建阶段 3 Plan 文档，锁定范围与接口
- 明确站点数据结构、配置存储方式、Nginx 模板策略
- 实现站点列表/新增/编辑/删除的后端接口
- 实现前端站点页基础表单与列表联调
- 明确“配置生成”和“配置生效”的错误处理

本周期产出：

- 阶段 3 设计文档
- 站点 CRUD 可跑通
- 基础站点列表页可演示

### 周期 3：反向代理生效链路与 Linux 验证

目标：把阶段 3 从“有 CRUD”推进到“真的能管理 Nginx”。

本周期工作：

- 完成反向代理目标配置、生成 Nginx 配置文件
- 增加配置校验、重载/回滚策略
- 在 Linux 环境完成一次站点创建与代理生效验证
- 完成阶段 3 文档、异常提示与验收说明

本周期产出：

- 阶段 3 MVP 完成
- 可验证的 Nginx 生效链路

### 周期 4：数据库管理 MVP

目标：完成课程与 MVP 所需的数据库管理最小闭环。

本周期工作：

- 新建阶段 4 Plan 文档，拆分“数据库管理”和“计划任务”
- 先做 MySQL 库/用户列表、创建、删除
- 前端数据库页从占位切换为可操作页面
- 增加危险操作确认、错误提示与基础校验
- 完成最小验收脚本与文档

本周期产出：

- 阶段 4 数据库管理 MVP 完成
- MVP 五大功能全部落地

### 周期 5：稳定性、部署与课程交付

目标：把“功能做出来”转换为“可以交付和答辩”。

本周期工作：

- 补齐关键路径测试
- 整理部署说明与演示步骤
- 收口默认配置、安全项和环境变量
- 根据时间决定是否补计划任务、Docker 或多机扩展说明

本周期产出：

- 一套可演示、可部署、可答辩的版本
- 最终项目总结与阶段复盘

## 7. 当前周期建议

如果从今天开始进入下一轮开发，建议**当前周期直接定义为“周期 1：阶段 2 收口与验收准备”**，不要马上跳到阶段 3。

优先级建议：

1. 文档同步与阶段 2 收口
2. Linux 联调验证监控链路
3. 最小测试与安全默认值收口
4. 阶段 3 Plan 设计

原因：

- 当前已有代码已经足够支撑一次阶段性总结
- 但监控模块还没有完全收口，直接进入网站管理容易把技术债继续带大
- 阶段 3 会首次进入“真正系统配置变更”场景，必须先把基础工程质量提一档

---

## 8. 推荐验收口径

后续每个周期结束时，至少按以下标准判断是否真的完成：

- 有对应 Plan / 设计文档更新
- 前后端功能都能跑通
- 至少有一轮手工验收步骤
- Linux 目标环境完成关键路径验证
- 遗留问题有记录，不以“暂时能跑”视为完成
