# 网页端 SSH 终端 — 设计与实现总结

## 1. 目标与背景

在 PanelLab 的 Dashboard 中集成 Web 端 SSH 终端，使用户可直接在浏览器中对远程服务器进行 SSH 控制，无需在本地反复打开 Windows PowerShell 等终端工具，提升运维效率。

**实现状态**：已完成并集成到主流程。

---

## 2. 技术选型

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 UI | Vue 3 | 与现有面板一致 |
| 终端仿真 | @xterm/xterm | 在浏览器中渲染终端、处理键盘输入与 ANSI 输出 |
| 自适应 | @xterm/addon-fit | 窗口/容器尺寸变化时自动调整终端行列 |
| 通信 | 原生 WebSocket | 与后端保持长连接，双向传输终端 I/O |
| 后端 | FastAPI WebSocket | 处理 WS 连接、鉴权、与 SSH 桥接 |
| SSH 客户端 | paramiko | 建立 SSH 连接并调用 `invoke_shell()` 打开交互式 PTY |
| 配置与密码 | 用户中心「服务器库」+ utils.cipher | 每用户独立维护多台服务器，密码加密存储与解密 |

---

## 3. 架构与数据流

### 3.1 连接建立

1. 用户登录后进入「终端控制」页；若尚无终端会话且服务器库非空，可自动创建第一个会话，或由用户点击「+ 新建终端」从**用户中心 → 服务器数据中心**维护的服务器库中选择一台服务器。
2. 前端从本地存储读取 JWT Token，建立 WebSocket：  
   `ws://<host>/api/ws/terminal?token=<jwt>&server_id=<服务器库中的服务器ID>`
3. 后端接受连接后：
   - 用 Query 中的 `token` 校验 JWT 并查询用户；
   - 根据 `server_id` 从**服务器库**（`Server` 表，按 `user_id` 隔离）读取对应服务器配置；
   - 用 `decrypt_password` 解密密码，再使用 `paramiko.SSHClient` 连接该主机并 `invoke_shell(term="xterm")`。

### 3.2 数据流转

- **前端 → 后端**：xterm 的 `onData` 将用户按键通过 WebSocket 以文本发送；后端收到后写入 SSH Channel。
- **后端 → 前端**：后台异步循环从 SSH Channel `recv()` 读数据，解码为 UTF-8 后通过 WebSocket 发给前端，由 xterm 渲染。
- **Resize**：前端用 ResizeObserver 监听容器尺寸变化，`FitAddon.fit()` 后向 WebSocket 发送 `{"type":"resize","cols":N,"rows":M}`；后端调用 `channel.resize_pty(width=N, height=M)`。

### 3.3 断开与资源释放

- 用户关闭某个终端标签（×）或登出 → 对应 WebSocket 关闭；后端在 `ws_to_ssh` / `ssh_to_ws` 中捕获异常或循环结束，在 `finally` 中 `client.close()` 释放 paramiko 连接。
- 从终端页跳转到其他页面**不关闭** WebSocket，连接由全局 store 保持；返回终端页时继续使用原会话。

## 4. 接口与依赖

### 4.1 WebSocket 接口

- **路径**：`/api/ws/terminal`
- **查询参数**：`token`（必填，JWT）、`server_id`（必填，服务器库中的服务器 ID）。
- **鉴权**：未传或无效 token 时发送错误提示并 `close(1008)`；`server_id` 不存在或不属于当前用户、解密失败时同样关闭并提示。

### 4.2 配置来源

- 远程连接信息来自**用户中心 → 服务器数据中心**维护的**服务器库**（表 `Server`，按 `user_id` 隔离）。
- 支持每用户多台服务器；新建终端时从该列表选择，标签以所选服务器 IP（或主机名）展示。

### 4.3 运行环境

- **后端**：Python 3.7+（推荐 3.8+）。SSH 连接在异步上下文中通过 `loop.run_in_executor(None, _do_connect)` 执行，避免阻塞事件循环，并兼容不提供 `asyncio.to_thread` 的 Python 3.8 及以下。
- **前端**：需支持 WebSocket 与 ES Module（Vite 构建）。开发时 Vite 代理需对 `/api` 开启 `ws: true` 以转发 WebSocket。

## 5. 实现总结（已完成的开发内容）

### 5.1 后端

- **终端 WebSocket**：`backend/routers/terminal.py`
  - 路由 `/api/ws/terminal`，Query 参数 `token`、`server_id`；从**服务器库**（`Server` 表，按 `user_id` 隔离）取连接信息并解密；
  - 使用 `paramiko` 建立 SSH、`invoke_shell()`，双任务 `ws_to_ssh` / `ssh_to_ws`；支持 `{"type":"resize", "cols", "rows"}` 调用 `channel.resize_pty`；
  - 服务关闭时捕获 `asyncio.CancelledError`，避免控制台刷屏。
- **服务器库 API**：`backend/routers/servers.py`
  - `GET/POST/PUT/DELETE /api/servers`，每用户独立增删改查；创建/更新时密码经 `utils.cipher` 加密存储。
- **注册**：`backend/main.py` 中 `include_router(terminal.router)`、`include_router(servers.router)`。

### 5.2 前端

- **页面**：`frontend/src/views/dashboard/Terminal.vue`
  - 使用 @xterm/xterm + FitAddon；**无**「连接/断开」按钮，改为**多终端标签栏** +「+ 新建终端」从服务器库选择；
  - 标签以**服务器 IP** 命名，带状态点（绿/黄/红）；关闭按钮（×）悬停显示；当前标签高亮（蓝色底边）；
  - WebSocket 由全局 `useTerminalStore` 管理，路由切换不关闭连接；登出时 `closeAllSessions()`；
  - 终端**容器 DOM 常驻**，无会话时显示占位层（不透明背景 + 提示文案），避免新建终端时容器重建导致 xterm 未挂载；无会话时对 xterm 执行 `term.clear()`，避免残留 shell 提示符。
- **状态**：`frontend/src/composables/useTerminalStore.js` — 会话列表、当前会话、createSession/closeSession/attach/detach/sendToSession，会话带 buffer 便于切换回放。
- **用户中心**：`frontend/src/views/dashboard/Settings.vue` — 左侧导航置顶「用户中心」，可折叠区块中包含「服务器数据中心」表格与新建/编辑/删除服务器表单。
- **路由与导航**：`/terminal` 对应 `Terminal.vue`；`DashboardLayout.vue` 中「终端控制」入口；登录/注册成功后跳转至「用户中心」。
- **代理**：`frontend/vite.config.js` 中 `/api` 配置 `ws: true`。

### 5.3 依赖

- 前端：`@xterm/xterm`、`@xterm/addon-fit`（见 `frontend/package.json`）。
- 后端：`paramiko`（已有）、FastAPI WebSocket、数据库与 `utils.cipher`（已有）。

## 6. 使用说明（用户侧）

- **使用前**：在**用户中心 → 服务器数据中心**中至少添加一台服务器（名称、IP/主机名、端口、用户名、密码）；终端从该列表选择连接目标。
- **连接**：进入「终端控制」后，若无会话且服务器库非空可自动打开第一个服务器终端；或点击「+ 新建终端」从列表选择一台服务器建立新会话。标签显示该服务器 IP，绿点表示已连接。
- **多终端**：可多次点击「+ 新建终端」连接多台（或同一台）服务器，通过标签切换；关闭某标签即断开该会话。
- **持久连接**：切换到其他菜单再返回终端页，连接保持，无需重连。
- **常用指令与历史**：右侧栏提供常用指令（系统/网络/Docker/日志等）与历史指令，点击即向当前选中终端发送命令。
- 若提示「请先在用户中心 → 服务器数据中心添加服务器」，请先在用户中心添加至少一台服务器。

## 7. 故障排查与已知约束

| 现象 | 可能原因 | 处理建议 |
|------|----------|----------|
| `module 'asyncio' has no attribute 'to_thread'` | 后端运行在 Python 3.8 或更早 | 已改为 `run_in_executor`，兼容 3.7+；若仍报错请确认未使用旧版 `terminal.py` |
| WebSocket 连接失败 / 立即断开 | 未登录、token 过期、或代理未转发 WS | 确认已登录、Vite 代理配置 `ws: true`，生产环境 Nginx 需正确代理 `/api` 的 WebSocket |
| SSH Connection failed | 网络不通、远程未开 SSH、账号密码错误、防火墙 | 在本地 PowerShell 用相同账号测试 SSH；检查后端日志中的异常信息 |
| 终端无输出或乱码 | 编码或 PTY 类型 | 当前后端以 UTF-8 解码 SSH 输出；若远程为其他 locale，可考虑在远程设置 `LANG=en_US.UTF-8` 等 |

**当前限制**：

- 终端连接数无硬性上限，建议根据实际负载控制同时打开的会话数。

## 9. 已实现：持久连接与多终端会话（设计依据）

> 以下为设计时的目标与约定，**已按此实现**；当前行为以第 3～6 节及下文 10 节为准。

### 9.1 问题与目标

- **问题 1：连接随页面销毁而丢失**
  - 当前终端连接与 xterm 实例都绑定在 `Terminal.vue` 内部组件状态中；
  - 当用户从「终端控制」跳转到其他页面时，组件被卸载，WebSocket 被关闭，SSH 会话结束；
  - 再次返回终端页时需要重新连接，且**看不到之前的终端输出与使用记录**。
- **问题 2：手动「连接 / 断开」按钮影响体验**
  - 目前需要用户显式点击「连接」与「断开」，对单机运维场景来说偏“工具思维”，不符合「面板一直在线、终端随时可用」的体验预期。
- **问题 3：无法多开终端**
  - 现有设计仅支持单个会话；在多个任务（查看日志 + 部署 + 监控进程）并行时不够方便。

**目标：**

1. **持久连接**：在整个仪表盘使用期间，SSH 连接尽量保持，路由跳转不打断；仅在用户登出、显式关闭所有终端或发生严重错误时才释放。
2. **自动连接**：在用户进入仪表盘/终端相关功能时自动建立 SSH 连接，无需手动点「连接」按钮。
3. **多会话**：支持**多终端会话并行**（“终端多开”），每个会话独立的 Shell 与滚动区，用户可在前端切换与关闭。
4. **安全与资源可控**：限制单用户最大会话数（如 3～5），异常断开时自动标记状态并可一键重连。

### 9.2 前端架构调整：终端服务单例

为实现「持久连接 + 多会话」，前端需要将终端连接逻辑从 `Terminal.vue` 提升到全局服务（或 Pinia store）：

- **终端服务（示意命名：`useTerminalService`）职责：**
  - 持有**全局唯一**的 WebSocket 连接管理器；
  - 管理一个会话列表 `sessions: TerminalSession[]`，每个会话包含：
    - `id`: 唯一标识（UUID 或自增）；
    - `serverId`: 所连接服务器在「服务器库」中的 ID；
    - `host`（或等价字段）：连接目标 IP/主机名，用于**标签展示**（见 9.4）；
    - `status`: `connecting` / `connected` / `disconnected` / `error`；
    - `ws`: 与该会话绑定的 WebSocket 实例；
    - `buffer`: 可选，最近 N 行输出缓存（用于重新挂载时回放）。
  - 提供对外 API：
    - `createSession(serverId)` / `closeSession(id)` / `closeAllSessions()`（新建会话时需指定要连接的服务器）；
    - `send(id, data)`：向指定会话发送数据（按键、命令）；
    - `attachXterm(id, termInstance)`：将某个会话与当前 xterm 实例绑定（`Terminal.vue` 中只做 attach/detach）。
- **`Terminal.vue` 的角色变化：**
  - 不再直接创建与维护 WebSocket；
  - 在 `onMounted` 时：
    - 若尚无会话，可引导用户从服务器库选择一台服务器后调用 `createSession(serverId)`，或保留“空状态”由用户点击「新建终端」再选服务器；
    - 将当前选中会话 `attachXterm` 到页面上的 xterm 容器；
    - 若会话已有历史 `buffer`，可选择回放到 xterm。
  - 在 `onBeforeUnmount` 时：
    - 仅做 `detach`（断开 xterm 与会话输出绑定），**不关闭 WebSocket / SSH 会话**。

> 这样，路由切换只影响 UI，不影响后台终端连接；用户返回终端页时能看到当前会话的持续输出与状态。

### 9.3 连接生命周期与自动连接策略

**触发时机：**

- 用户登录并进入 Dashboard 后：
  - 可以选择两种策略之一（实现时在代码与配置中明确）：
    1. **懒加载策略（推荐）**：当用户**首次进入「终端控制」页**或首次点击「新建终端」时，从**服务器库**选择一台服务器，再创建 SSH 会话并保持其生命周期；
    2. **激进策略**：在仪表盘 Layout 初始化时即创建默认会话（需从服务器库取默认或第一条服务器；可能增加资源占用）。
- 无论策略如何，一旦会话创建并连接成功，在用户保持登录状态期间：
  - 切换到监控、站点、数据库等页面时**不关闭**现有会话；
  - 返回「终端控制」即可继续当前会话。

**连接结束时机：**

- 用户登出（清除 token）时：
  - 自动调用 `closeAllSessions()`，关闭所有 WebSocket 与 SSH 连接，释放资源。
- 用户在终端页通过 UI 主动「关闭全部终端」或关闭单个标签页：
  - 关闭对应会话，后端感知到 WebSocket 关闭后也释放 SSH。
- 异常情况：
  - 网络中断、后端异常关闭连接等，前端会话状态置为 `error` 或 `disconnected`，提供「重新连接」操作。

**空闲连接策略（可选）：**

- 可增加简单空闲策略（后续实现时再细化）：如会话 N 分钟无输入，则在前端弹出提示或自动关闭；当前设计阶段暂不强制。

### 9.4 多终端会话（多开）设计

**会话模型：**

- 支持每个用户在一个浏览器会话中打开多条终端会话（默认上限可设为 3～5 条，超出时给出提示）。
- 每个会话**从「服务器库」中选择一台服务器**进行连接；可同时打开多台不同服务器的终端，或同一台服务器的多个 Shell（视实现而定）。

**终端标签命名（重要）：**

- 标签**不以「终端 1」「终端 2」命名**，而**以当前连接的服务器的 IP（或主机名）作为标签名**，更利于区分多台机器。
- 示例：`192.168.1.100`、`43.106.25.171`；若服务器库中存了主机名且优先展示，也可显示主机名（如 `db-server`），但 IP 更通用、建议作为默认展示。

**前端 UI（初版建议）：**

- 将现有的「连接 / 断开」按钮调整为：
  - 顶部增加**会话标签栏**：
    - 类似浏览器标签页，每个标签显示**该会话所连接服务器的 IP**（如 `43.106.25.171`），同一台服务器多开时可带后缀区分（如 `43.106.25.171 (2)`），具体规则实现时定；
    - 当前标签高亮，显示对应会话的输出与状态；
    - 每个标签右侧有 `×` 关闭按钮（关闭单个会话）。
  - 在标签栏最右侧提供「+ 新建终端」按钮：
    - 点击后**从服务器库中选择要连接的服务器**（下拉或弹窗），选定后调用 `createSession(serverId)` 建立新会话并切换到该会话；
    - 不需要用户手动点击「连接」。
- 连接状态反馈：
  - 标签标题旁用小点或文字标记状态：如灰色「未连接」、黄色「连接中」、绿色「已连接」、红色「错误」；
  - 发生错误时，标签中可出现「重连」小按钮。

**历史与常用指令栏：**

- 保持当前右侧「常用指令 + 历史指令」设计：
  - 常用指令：对**当前选中会话**发送命令；
  - 历史指令：全局共享或与用户绑定，不随会话关闭而清空（保持当前实现思路）。

### 9.5 服务器库与用户中心（配置来源）

- **终端连接目标不再依赖「系统监控」的单一远程配置**，改为**根据「服务器库」选择**：
  - 用户在终端中连接的服务器，均来自其个人**服务器库**（即用户可维护的多台服务器列表）；
  - 新建终端时，从该列表中选择一台服务器建立 SSH 会话；标签名使用所选服务器的 IP（见 9.4）。
- **服务器库的落地顺序建议：**
  - 可在**用户中心**先实现「服务器库」功能：存储当前用户的多个服务器信息（如名称、IP、端口、用户名、密码加密存储等），支持增删改查；
  - 终端增强（持久连接 + 多开 + 按服务器库选择）依赖该数据源，实现时调用同一套服务器列表与鉴权。
- **与系统监控的关系（可选）：**
  - 若产品上希望「系统监控」仍使用独立配置，可保留现有 `monitor_remote_config` 仅给监控用；
  - 终端与服务器库对接后，不再依赖 `monitor_remote_config`；若后续需要监控与终端共用同一批服务器，可在用户中心/服务器库中统一维护，监控再从服务器库选择目标。

### 9.6 实现注意事项与迁移

- **实现顺序建议：**
  - **先做用户中心「服务器库」**：在用户中心增加服务器管理（多台服务器的增删改查、密码加密存储等），终端增强依赖该数据源；若已有类似表（如 `servers`）可复用并暴露 API。
  - **再做终端持久连接与多开**：终端服务、按服务器库选择、标签以 IP 展示等，按 9.2～9.5 实施。
- **兼容现有实现：**
  - 当前 `Terminal.vue` 中的 WebSocket 与 xterm 初始化流程将在实现阶段迁移到终端服务中；
  - 为避免一次性重构风险，可先实现终端服务并让 `Terminal.vue` 逐步从“持有连接”变为“仅负责展示”。
- **资源控制：**
  - 单用户会话上限需要在前后端同时考虑（前端限制创建数量，后端可在需要时增加保护）。
- **文档与用户手册**：已同步更新本设计文档第 3、5、6 节及用户手册「远程终端」与「用户中心」说明。

## 10. 实现细节（终端容器与占位）

- **容器常驻**：终端页中用于挂载 xterm 的 DOM 容器（`terminalRef`）**始终存在**，不随「无会话」而移除。这样在用户关掉所有终端后再点击「新建终端」时，xterm 仍能挂到同一容器上，无需依赖路由重新挂载。
- **占位层**：当无当前会话（`!currentSession`）时，在容器上叠加一层**占位提示**（「点击『新建终端』从服务器库选择一台服务器连接」）；该层使用不透明背景（与终端区域同色），避免下层 xterm 残留内容（如 shell 提示符）透出。
- **无会话时清空**：在「当前会话」变为空时（关闭最后一个标签），除在 `currentSessionId` 的 watch 中调用 `term.clear()` 外，另设 watch 监听 `sessions.length === 0`，在 `nextTick` 中再次执行 `term.clear()`，确保画布无残留文字。
