# 手机验证码登录：技术实现与资源准备

本文档说明如何实现**手机验证码登录**，包括技术实现要点与**资源准备**（以阿里云短信服务为例的购买与配置）。与 [design-login-multi.md](design-login-multi.md) 中「手机验证码」部分衔接：当前项目已具备 mock 发码与验证码登录接口，本文档描述如何改为真实短信发送及上线前需完成的准备工作。

---

## 1. 概述与目标

| 项目 | 说明 |
|------|------|
| 现状 | 后端已有 `POST /api/auth/sms/send`、`POST /api/auth/sms/login`，验证码为 mock（固定 123456），节流 60s、有效期 5 分钟。 |
| 目标 | 将「发送验证码」改为通过**阿里云短信服务**真实下发；资源准备包括开通服务、申请签名与模板、获取 AccessKey 等。 |
| 范围 | 技术实现（流程、存储、安全、对接阿里云 API）；阿里云控制台操作与计费说明。 |

---

## 2. 技术实现

### 2.1 整体流程

```
用户输入手机号 → 前端请求「发送验证码」→ 后端校验手机号、节流
    → 生成 6 位随机码、写入存储、调用短信服务发送
    → 用户输入验证码并提交
    → 后端校验验证码与过期时间 → 按手机号查/建用户 → 签发 JWT 返回
```

- **发送**：同一手机号 60 秒内仅可发送一次；验证码 6 位数字，有效期 5 分钟。
- **登录**：验证码正确且未过期则通过；若该手机号无对应用户则自动创建（`username=phone_xxx`），与 [design-login-multi.md](design-login-multi.md) 一致。

### 2.2 接口约定（与现有一致）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/auth/sms/send | 请求体 `{ "phone": "13800138000" }`；校验格式与节流；发送验证码；成功返回 `{ "message": "验证码已发送" }`。 |
| POST | /api/auth/sms/login | 请求体 `{ "phone": "13800138000", "code": "123456" }`；校验通过后返回 `{ token, user: { username } }`。 |

手机号：11 位大陆手机（`1` 开头）；验证码：6 位数字。

### 2.3 验证码存储与节流

- **存储**：键为手机号，值为 `{ code, expires_at }`。当前实现为内存字典；多实例部署时建议改为 **Redis**，键如 `sms:code:{phone}`，设置 TTL 与节流键 `sms:throttle:{phone}`。
- **节流**：按手机号限制发送频率（如 60s 内仅可发送一次），防止恶意刷短信与成本激增。

### 2.4 安全要点

- 不向前端返回真实验证码；生产环境关闭「固定验证码」mock。
- 验证码一次性使用：校验通过后立即从存储中删除。
- 接口可配合图形验证码或设备指纹做防刷（可选，后续扩展）。
- AccessKey、签名、模板等敏感信息仅放在服务端环境变量，不写进代码仓库。

### 2.5 后端对接阿里云短信（代码改造要点）

在保留「mock」模式的前提下，增加「阿里云」真实发送：

1. **配置项**（见下方 4. 配置）：  
   `SMS_PROVIDER=mock|aliyun`，阿里云时需配置 `ALIYUN_ACCESS_KEY_ID`、`ALIYUN_ACCESS_KEY_SECRET`、`ALIYUN_SMS_SIGN_NAME`、`ALIYUN_SMS_TEMPLATE_CODE` 等。
2. **发送逻辑**：  
   - `SMS_PROVIDER=mock`：与现有一致，生成固定或随机 6 位码写入存储，不调用外部 API。  
   - `SMS_PROVIDER=aliyun`：生成 6 位随机码写入存储后，调用阿里云 SendSms 接口，将模板变量（如 `code`）传入；发送失败时不写入存储或写入后标记失败，并返回明确错误信息（如 400/502）。
3. **阿里云 API**：  
   - 接口：`SendSms`（产品为「短信服务」）。  
   - 必填参数：`PhoneNumbers`、`SignName`、`TemplateCode`；`TemplateParam` 为 JSON 字符串，如 `{"code":"123456"}`（变量名需与控制台模板中一致，常见为 `code`）。  
   - 推荐使用官方 Python SDK：`alibabacloud_dysmsapi20170525`，或使用 OpenAPI 的 HTTP 调用方式。
4. **依赖**：  
   - 若用 SDK：在 `requirements.txt` 增加 `alibabacloud_dysmsapi20170525`、`alibabacloud_tea_openapi`（按 SDK 文档为准）。  
   - 若用 HTTP：使用 `httpx` 等按阿里云 OpenAPI 签名规范请求即可。

这样，开发/测试继续用 mock；生产配置阿里云后即可真实发码，无需改接口契约。

## 3. 资源准备：阿里云短信服务（以阿里云为例）

以下为使用**阿里云「短信服务」（SMS）** 发送验证码的完整准备步骤，包括开通、签名、模板、权限与计费。

### 3.1 前提条件

- 已注册 [阿里云账号](https://www.aliyun.com/)。
- 完成**实名认证**（个人或企业均可；企业认证后可使用更多模板类型）。
- 若发往**中国大陆**，需关注当前政策：部分新账号可能限制「发往中国大陆」开通，仅历史已开通账户可继续使用；具体以控制台为准。若无法开通大陆短信，可先使用「短信认证服务」或仅做 mock 开发。

### 3.2 开通短信服务

1. 登录 [阿里云控制台](https://home.console.aliyun.com/)。
2. 进入 **短信服务** 产品页，或直接打开 [短信服务控制台](https://sms.console.aliyun.com/)（如 `https://dysms.console.aliyun.com/`，以当前阿里云控制台实际路径为准）。
3. 若未开通，按页面提示点击「**开通短信服务**」并同意《短信服务开通协议》。
4. 开通后，在控制台可看到「概览」「国内消息」等菜单。

### 3.3 申请短信签名

短信签名是展示在用户手机上的「发送方」标识（如您的产品名、公司名）。

1. 在短信服务控制台，进入「**国内消息**」→「**短信签名**」。
2. 点击「**添加签名**」/「**创建签名**」。
3. 填写：
   - **签名名称**：如「PanelLab」「某某科技」等，需与您的业务一致。
   - **签名来源**：根据实际选择（如「公司」、「网站」、「App」等）。
   - **申请说明**：简要说明用途，例如「用于 PanelLab 产品登录验证码发送」。
4. 提交后等待审核，一般 **2 小时内**（工作时间）完成；审核通过后即可在「发送短信」时选择该签名。  
   **注意**：签名创建后通常不可修改，只能删除后重新申请；名称需符合规范，避免使用敏感词。

### 3.4 申请短信模板（验证码）

验证码短信需使用「验证码」类模板，模板中通过变量占位验证码内容。

1. 在控制台进入「**国内消息**」→「**短信模板**」。
2. 点击「**添加模板**」/「**创建模板**」。
3. 选择模板类型：**验证码**。
4. 填写模板内容，例如：
   - 模板内容：`您的验证码为：${code}，5分钟内有效，请勿泄露。`
   - 变量名需与代码中 `TemplateParam` 一致，常见为 `code`；若使用其他变量名（如 `verification_code`），则代码中需传对应 JSON 键。
5. 填写「使用说明」等必填项后提交审核，同样约 **2 小时内** 完成。
6. 审核通过后，在模板列表中复制 **模板 CODE**（形如 `SMS_123456789`），后续配置到后端环境变量 `ALIYUN_SMS_TEMPLATE_CODE`。

### 3.5 创建 RAM 用户与 AccessKey（推荐）

为安全起见，不要使用主账号的 AccessKey，而是为短信服务单独创建 **RAM 子用户** 并仅授予短信权限。

1. 登录 [RAM 控制台](https://ram.console.aliyun.com/)。
2. 创建用户：**身份管理** → **用户** → **创建用户**；登录名可如 `sms-sender`，仅勾选「OpenAPI 调用」。
3. 为该用户添加权限：在权限策略中选择 **AliyunDysmsFullAccess**（短信服务完全访问），或自定义最小权限策略仅包含 `dysms:SendSms`。
4. 创建 **AccessKey**：在该用户详情中「创建 AccessKey」，保存 **AccessKey ID** 和 **AccessKey Secret**（Secret 仅展示一次，请妥善保存）。
5. 将 AccessKey ID 与 Secret 配置到后端环境变量（见下文 4），不要提交到代码仓库。

### 3.6 计费与限频

- **计费**：按条计费，验证码类约 **0.045 元/条** 起（以阿里云当前价格为准）；发送失败一般不扣费。可购买套餐包或按量付费。
- **限频**（阿里云侧）：单号码通常限制为 1 条/分钟、5 条/小时、10 条/天等，具体以控制台与文档为准；本系统已在应用层做 60 秒节流，与平台限频互补。

完成上述步骤后，您将拥有：**签名名称**、**模板 CODE**、**AccessKey ID**、**AccessKey Secret**，即可在后端配置并启用真实发送。

## 4. 环境变量与配置示例

在 `backend/.env` 或部署环境中增加或修改如下变量（生产环境务必使用强 Secret 且不提交到 Git）：

```env
# 短信：mock | aliyun（先 mock 开发，上线前改为 aliyun）
SMS_PROVIDER=mock

# 阿里云短信（仅当 SMS_PROVIDER=aliyun 时使用）
ALIYUN_ACCESS_KEY_ID=你的AccessKeyId
ALIYUN_ACCESS_KEY_SECRET=你的AccessKeySecret
ALIYUN_SMS_SIGN_NAME=你在控制台申请的签名名称
ALIYUN_SMS_TEMPLATE_CODE=SMS_123456789
# 若模板变量名不是 code，可增加：ALIYUN_SMS_TEMPLATE_PARAM_CODE=verification_code（变量名）
```

后端读取方式示例（与现有 `config.py` 风格一致）：

- `SMS_PROVIDER`：默认 `mock`。
- `ALIYUN_ACCESS_KEY_ID`、`ALIYUN_ACCESS_KEY_SECRET`：字符串，空表示未配置。
- `ALIYUN_SMS_SIGN_NAME`、`ALIYUN_SMS_TEMPLATE_CODE`：字符串；对接阿里云时必填。

发送逻辑伪代码：

- 若 `SMS_PROVIDER != "aliyun"`：按现有 mock 逻辑生成验证码并写入存储，返回成功。
- 若 `SMS_PROVIDER == "aliyun"`：生成 6 位随机码，写入存储；调用阿里云 SendSms（PhoneNumbers=手机号，SignName、TemplateCode 从配置读，TemplateParam=`{"code": "随机码"}`）；若调用失败则返回 502 或 400 及明确提示，且可不写入/可撤销刚写入的验证码（视业务选择）。

## 5. 测试与上线注意

- **开发/测试**：保持 `SMS_PROVIDER=mock`，前端与自动化测试使用固定或可预测验证码（如 123456），无需消耗真实短信。
- **联调**：在测试环境配置阿里云参数，使用真实手机号少量发送，确认能收到且内容正确；注意控制台有日发送量等限制。
- **上线**：生产环境改为 `SMS_PROVIDER=aliyun`，确认签名、模板均已审核通过，AccessKey 使用 RAM 子账号并限制权限；监控发送成功率与成本。
- **降级**：若阿里云暂时不可用，可考虑在代码中捕获异常并返回「短信服务暂时不可用，请稍后重试」，或临时切回 mock（仅限紧急情况且注意安全）。
