# 数据库与计划任务：需求与设计（阶段 4）

本文档为阶段 4「数据库与计划任务」的 Plan，格式与 [docs/README.md](../README.md) 中「Plan 格式说明」一致。

---

## 1. 目标与范围

| 项目 | 说明 |
|------|------|
| 目标 | 在仪表盘「数据库管理」页完成 MySQL 基础管理闭环（库/用户列表、创建、删除），并给出计划任务（cron）的最小可落地方案。 |
| 本阶段范围 | **包含**：数据库与用户的基础 CRUD、危险操作确认、执行结果反馈、基础权限与参数校验；**不包含**：SQL 在线编辑器、跨实例复制、备份恢复平台化、复杂任务编排。 |
| 运行环境 | 目标运行环境为 **Linux + MySQL**；Windows 开发阶段可先完成前后端接口与交互，最终必须在 Linux 上验证真实命令执行。 |
| 依赖/前置 | 依赖阶段 1 登录鉴权、阶段 3 页面风格规范与错误反馈模式。若需在目标机执行系统命令，需确认运行账号具备最小必要权限。 |

## 2. 功能需求

### 2.1 必须功能（MVP）

| 功能 | 说明 |
|------|------|
| 数据库列表 | 返回可管理数据库列表（可按业务约定仅展示非系统库），含创建时间等基础信息。 |
| 创建数据库 | 支持创建 MySQL 数据库，字符集默认 `utf8mb4`。 |
| 删除数据库 | 支持删除数据库（默认要求二次确认，提示不可逆）。 |
| 用户列表 | 返回数据库用户列表（至少包含用户名、主机、权限摘要）。 |
| 创建用户并授权 | 支持创建用户并授予指定数据库权限（默认最小可用权限策略）。 |
| 删除用户 | 支持删除数据库用户并提示风险。 |
| 错误反馈 | 连接失败、权限不足、SQL 执行失败等错误必须明确返回给前端。 |

### 2.2 阶段内可选（建议）

- 数据库详情页（库大小、表数量统计）
- 用户权限编辑（变更授权）
- 连接可用性检测按钮（一键校验 MySQL 连接）

### 2.3 计划任务（cron）范围约束

本阶段只做“可选预研 + 结构预留”，不强制进入 MVP：

- 给出计划任务的数据结构与接口草案
- 预留前端入口与占位提示
- 明确下一阶段可直接接续的实现路径

## 3. 接口约定

所有接口均要求登录（Bearer JWT），统一前缀为 `/api/db`。

### 3.1 数据库管理

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/db/databases | 返回数据库列表。 |
| POST | /api/db/databases | 创建数据库。 |
| DELETE | /api/db/databases/{name} | 删除数据库。 |

**创建数据库请求示例：**

```json
{
  "name": "panel_app",
  "charset": "utf8mb4",
  "collation": "utf8mb4_unicode_ci"
}
```

### 3.2 用户管理

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/db/users | 返回用户列表。 |
| POST | /api/db/users | 创建数据库用户并授权。 |
| DELETE | /api/db/users/{username} | 删除用户（可通过 query 指定 host）。 |

**创建用户请求示例：**

```json
{
  "username": "panel_reader",
  "password": "StrongPassword_123",
  "host": "%",
  "database": "panel_app",
  "privileges": ["SELECT", "INSERT", "UPDATE", "DELETE"]
}
```

### 3.3 计划任务（预留接口）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/db/jobs | 返回计划任务列表（预留，可返回空实现）。 |
| POST | /api/db/jobs | 新建计划任务（预留）。 |
| DELETE | /api/db/jobs/{id} | 删除计划任务（预留）。 |

### 3.4 错误约定

- `400`：参数非法（名称不合法、密码过弱、权限列表为空）
- `401`：未登录或 token 无效
- `403`：当前用户无操作权限
- `404`：目标数据库或用户不存在
- `409`：数据库名或用户名冲突
- `500`：服务端内部错误
- `503`：MySQL 不可用或命令执行失败

## 4. 数据与存储

### 4.1 阶段四数据策略

阶段四以“管理已有 MySQL 实例”为主，业务数据主要来自 MySQL 元数据，不强制新增业务表。

### 4.2 计划任务预留表（可选）

若决定在本阶段先落库，可新增 `scheduled_jobs`：

| 字段 | 说明 |
|------|------|
| id | 主键 |
| name | 任务名称 |
| cron_expr | cron 表达式 |
| command | 执行命令 |
| enabled | 是否启用 |
| last_run_at | 最近执行时间 |
| last_status | 最近执行状态 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 4.3 配置来源

- 优先使用现有环境变量：`MYSQL_HOST`、`MYSQL_PORT`、`MYSQL_USER`、`MYSQL_PASSWORD`、`MYSQL_DATABASE`
- 明确区分“业务连接库”和“管理操作对象”，避免误删当前业务核心库

## 5. 后端实现要点

### 5.1 模块划分建议

- `routers/db_admin.py`：数据库与用户接口
- `schemas/db_admin.py`：请求/响应模型
- `services/mysql_admin.py`：MySQL 管理命令与 SQL 执行封装
- （可选）`routers/jobs.py` + `services/cron_manager.py`：计划任务预留

### 5.2 实现策略

- 优先采用参数化 SQL，避免字符串拼接导致注入风险
- 将“SQL 执行/异常映射”集中在 service 层，router 仅做入参校验与响应转换
- 统一返回可读错误信息，便于前端直接提示

### 5.3 安全注意点

- 严格限制可操作对象：默认禁止删除系统库（如 `mysql`、`information_schema`）
- 用户名/库名采用白名单字符校验（字母、数字、下划线）
- 危险操作必须要求前端显式确认（例如 `confirm_text`）
- 避免日志打印明文密码

## 6. 前端实现要点

### 6.1 页面结构（`/databases`）

至少包含：

- 数据库列表卡片 + 创建按钮
- 数据库用户列表卡片 + 创建按钮
- 新建弹层（数据库/用户）
- 删除二次确认弹窗
- 统一成功/错误提示

### 6.2 交互规范

- 沿用阶段 3 的交互模式：列表加载态、空态、提交禁用态
- 危险操作按钮使用高风险视觉样式
- 后端返回 `detail` 时优先展示 `detail`

### 6.3 与现有风格一致性

- 保持与 `Sites.vue`、`Monitor.vue` 相同的卡片、按钮、表格体系
- 避免在阶段四引入新的大型 UI 依赖
