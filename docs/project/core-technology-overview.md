# PanelLab：主要功能核心技术讲解

本文档用于从技术实现角度解释 PanelLab 各主要功能的核心思路，帮助后续开发、答辩展示和阶段规划对齐。

阅读建议：

- 想了解“项目整体怎么工作”，先看第 1 节和第 2 节
- 想了解“当前已做了什么”，重点看用户体系、仪表盘、系统监控
- 想了解“后续怎么做”，重点看网站与反向代理、数据库管理

---

## 1. 项目整体技术结构

PanelLab 当前采用典型的前后端分离结构：

- **前端**：Vue 3 + Vue Router + Vite
- **后端**：FastAPI
- **数据库**：MySQL
- **鉴权方式**：JWT
- **运维侧能力**：通过 Python 直接读取系统信息，或通过 SSH 连接远程 Linux 主机执行采集逻辑

整体链路可以理解为：

1. 用户在前端页面发起登录或业务操作。
2. 前端通过 HTTP 请求访问 FastAPI 接口。
3. 后端完成鉴权、数据库读写、系统调用或远程连接。
4. 后端返回 JSON，前端再负责页面展示和交互反馈。

这样的结构有几个优点：

- 前后端职责清晰，便于分阶段推进
- 前端可以优先做页面骨架，后端接口逐步补齐
- 后端更适合承接系统监控、Nginx 管理、MySQL 管理等运维相关能力

## 2. 当前系统分层理解

从工程角度，当前项目大致分为 4 层：

### 2.1 展示层

主要在 `frontend/src/views` 与 `frontend/src/components`。

职责：

- 展示登录、注册、概览、监控等页面
- 负责表单输入、按钮操作、加载态和错误提示
- 负责轮询监控接口、切换监控目标

### 2.2 路由与鉴权层

主要在前端路由与后端依赖中。

职责：

- 前端根据是否有 token 控制访问受保护页面
- 后端根据 JWT 判断当前请求用户是否合法
- 将“登录态”从页面级逻辑提升为全局访问控制

### 2.3 业务接口层

主要在 `backend/routers`。

职责：

- 定义 `/api/auth/*`、`/api/monitor/*` 等业务接口
- 处理请求参数、权限校验和错误返回
- 协调数据库、系统采集、远程连接等具体能力

### 2.4 数据与系统能力层

主要在数据库模型、配置模块和工具模块中。

职责：

- 用户、远程监控配置等数据持久化
- 密码加密、环境变量读取
- 主机信息采集、远程 SSH 连接、后续 Nginx/MySQL 管理

## 3. 用户体系与登录鉴权

### 3.1 这一块解决什么问题

用户体系是整个面板的入口，它解决的是：

- 谁可以进入管理面板
- 请求是否来自合法用户
- 页面跳转时如何保持登录状态

没有这层，后面的监控、站点管理、数据库管理都无法安全开放。

### 3.2 当前实现思路

当前项目采用 **账号密码 + JWT** 的基础方案。

核心流程：

1. 用户在登录页输入用户名和密码。
2. 前端调用后端登录接口。
3. 后端校验用户名和密码是否正确。
4. 校验通过后生成 JWT token。
5. 前端保存 token，并在后续请求里放入 `Authorization: Bearer <token>`。
6. 后端在受保护接口中解析 token，得到当前用户。

### 3.3 为什么用 JWT

对当前项目来说，JWT 的优点比较直接：

- 不依赖服务端 session 存储，接口更简单
- 前后端分离项目接入方便
- 非常适合“先做单用户 / 小规模用户体系”的课程项目

需要注意的是，JWT 不是“更安全”，而是“更适合当前结构”。真正安全性仍取决于：

- `secret_key` 是否足够强
- token 是否过期、是否妥善保存
- 登录与敏感接口是否有进一步保护

### 3.4 这部分后续可扩展点

- 增加角色与权限，例如管理员、只读用户
- 增加第三方登录或短信登录
- 增加 token 过期刷新和更严格的前端校验

## 4. 仪表盘与页面骨架

### 4.1 为什么要先做骨架

仪表盘骨架的价值不在于“功能完成”，而在于先把产品的信息结构稳定下来：

- 有哪些一级模块
- 用户从哪里进入各能力页
- 后续功能接入时是否需要大改页面结构

因此阶段 1 先做布局和路由，是一个很合理的技术顺序。

### 4.2 当前实现方式

当前前端以 `DashboardLayout` 作为登录后的统一容器，子页面挂在它下面。

这样做的好处：

- 侧边栏、主题切换、用户区只写一份
- `/monitor`、`/sites`、`/databases` 可以逐步接入
- 后续新增计划任务、设置、多机管理时扩展成本更低

### 4.3 技术本质

仪表盘这一块本质上不是复杂业务逻辑，而是：

- 路由结构设计
- 信息架构设计
- 页面复用与状态隔离

也就是说，它是在给后续功能模块“搭舞台”。

## 5. 系统监控

### 5.1 系统监控解决什么问题

系统监控的核心目标是：让用户能在面板里快速看到当前主机或远程主机的运行状态。

对当前项目，最基础也最重要的四类指标是：

- CPU
- 内存
- 磁盘
- 网络

这些指标能覆盖大多数日常观察场景，例如：

- 机器是否负载过高
- 是否存在内存紧张
- 磁盘是否快满了
- 网络流量是否异常

### 5.2 为什么后端做监控采集

监控数据不能直接由前端采集，因为前端运行在浏览器里，没有权限读取服务器的真实系统状态。

因此必须由后端完成：

1. 读取系统信息
2. 转成统一 JSON 结构
3. 提供接口给前端展示

### 5.3 本机监控技术路线

当前本机监控使用 **psutil**。

`psutil` 的优势：

- Python 生态成熟，适合课程与原型项目
- 可以跨平台读取 CPU、内存、磁盘、网络等信息
- 接口简单，便于快速把监控能力做出来

本机监控的实现思路是：

1. 后端收到 `GET /api/monitor/stats`
2. 分别调用 CPU、内存、磁盘、网络采集函数
3. 组装成统一结构的 JSON
4. 返回给前端
5. 前端在监控页定时轮询并刷新卡片数据

这种方案属于**快照式监控**，特点是：

- 实现简单
- 不依赖时序数据库
- 适合做实时查看

但它也有边界：

- 没有历史数据
- 不适合长期趋势分析
- 不具备告警系统

### 5.4 远程监控技术路线

当前远程监控采用的是 **SSH + 远程执行采集脚本**，而不是在目标机上部署常驻 Agent。

这样设计的原因：

- 项目体量较小，先避免引入额外守护进程
- 实现成本低，适合阶段 2 快速做出差异化能力
- 前后端仍能复用与本机相同的数据结构和页面组件

具体过程：

1. 用户在前端配置远程 Linux 的 SSH 连接信息。
2. 后端把连接信息写入数据库，密码加密保存。
3. 当用户选择“远程服务器”时，后端通过 Paramiko 建立 SSH 连接。
4. 后端把一段 Python 采集脚本发送到远程主机执行。
5. 远程主机用 `psutil` 读取自身状态并输出 JSON。
6. 后端解析 JSON 后直接返回前端。

这套方案的关键点在于：**前端完全不需要知道数据来自本机还是远程，只关心返回的数据结构是否一致。**

### 5.5 监控方案的优点与风险

优点：

- 本机和远程可以复用同一套前端展示
- 不需要单独开发 Agent
- 适合阶段性快速实现

风险：

- 依赖远程主机已有 `python3` 和 `psutil`
- SSH 密码认证安全性一般
- 远程连接失败时稳定性和错误反馈要做得足够清楚
- 目前是轮询快照，不是完整监控平台

### 5.6 监控模块后续扩展方向

- 增加进程管理
- 增加短期历史曲线
- 增加多台主机管理
- 增加 SSH 密钥认证
- 增加采集预检查和安装引导

## 6. 网站与反向代理

> 本部分为阶段 3 的核心技术说明，属于“规划中的实现路线”。

### 6.1 这个功能本质上在做什么

“网站与反向代理”并不只是做一个站点列表页面，它本质上是在做两类能力：

- **站点管理**：让面板知道当前有哪些站点、它们监听什么域名、根目录在哪里、端口是什么
- **反向代理管理**：让面板能生成和更新 Nginx 配置，把外部请求转发到后端应用

这也是宝塔类面板最核心、最接近真实运维操作的模块之一。

### 6.2 为什么这块必须由后端驱动

反向代理管理涉及真实系统文件和系统命令，例如：

- 写入 Nginx 配置文件
- 校验配置是否正确
- 重载 Nginx 服务
- 处理失败回滚

这些操作浏览器无法完成，因此必须由后端负责。

### 6.3 推荐实现模型

建议把阶段 3 拆成两层：

#### A. 面板内部数据模型

先在数据库中维护一份“站点配置记录”，例如：

- 站点名称
- 域名
- 根目录
- 站点类型（静态站点 / 反向代理）
- 代理目标地址
- 监听端口
- 是否启用

这层的作用是：让前端和后端围绕统一数据模型完成 CRUD。

#### B. 系统实际生效层

再由后端根据数据库中的配置，生成 Nginx 配置文件并应用到系统中。

这意味着：

- 数据库记录是“面板视角的配置”
- Nginx 配置文件是“系统视角的配置”

两者不能混为一谈。

### 6.4 为什么建议“先 CRUD，后生效”

如果一开始就把“表单提交 = 直接改系统配置”绑在一起，调试会非常痛苦。

更稳妥的实现顺序是：

1. 先完成站点记录的新增、编辑、删除、列表
2. 再补“生成配置”
3. 再补“校验并生效”
4. 最后处理失败回滚和异常提示

这样做的好处：

- 便于前后端先联调业务结构
- 便于隔离“表单问题”和“系统配置问题”
- 更适合课程项目的渐进式开发

### 6.5 反向代理的技术核心

反向代理的核心不是“界面”，而是“配置生成”。

例如对于一个代理站点，面板最终要能生成类似这样的 Nginx 配置逻辑：

```nginx
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

也就是说，前端用户填写的是“业务参数”，后端真正生成的是“可被 Nginx 执行的配置文件”。

### 6.6 阶段 3 应重点处理的技术点

- 如何设计站点数据结构
- 配置文件写到哪里
- 如何避免覆盖系统已有配置
- 如何在执行 `nginx -t` 失败时回滚
- 如何把系统错误转成用户能理解的提示

### 6.7 这一模块的推荐边界

阶段 3 建议只做最小可行范围：

- 单机 Nginx
- HTTP 站点
- 单域名或简单域名输入
- 静态站点与反向代理两种模式
- 配置校验与重载

可以暂时不做：

- HTTPS 自动签证书
- 复杂 location 规则
- 多站点模板市场
- Docker 容器反代

## 7. 数据库管理

> 本部分为阶段 4 的核心技术说明，属于“规划中的实现路线”。

### 7.1 这个功能在管理什么

“数据库管理”至少包含两个对象：

- **数据库本身**
- **数据库用户及其权限**

如果只做“数据库列表”，那只是展示；真正有价值的地方在于：

- 创建数据库
- 删除数据库
- 创建数据库用户
- 给用户分配某个库的权限

### 7.2 为什么这块不能只靠 ORM

项目自己的业务数据可以用 SQLAlchemy 管理，但“数据库管理”功能本身是在操作 **MySQL 服务器**，这和管理自己项目表结构不是一回事。

也就是说，这个模块面对的是：

- `CREATE DATABASE`
- `DROP DATABASE`
- `CREATE USER`
- `GRANT ...`
- `SHOW DATABASES`

因此更适合：

- 由后端通过数据库连接执行管理语句
- 或者由后端封装专门的数据库管理服务层

### 7.3 推荐实现思路

建议把数据库管理也拆成两层：

#### A. 查询展示层

读取当前 MySQL 中已有的：

- 数据库列表
- 用户列表
- 基础元信息

#### B. 管理执行层

针对用户操作执行：

- 创建数据库
- 删除数据库
- 创建数据库用户
- 赋权或回收权限

### 7.4 这里的难点不在“页面”，而在“权限与安全”

数据库管理是高风险操作，因此必须格外关注：

- 是否允许删除系统库
- 是否允许删除正在使用的数据库
- 用户名、库名是否做合法性校验
- 错误是否能明确返回
- 是否需要二次确认

阶段 4 最少也要把“危险操作确认”做好。

### 7.5 阶段 4 的建议边界

建议先只支持 MySQL 的最小能力：

- 数据库列表
- 创建数据库
- 删除数据库
- 基础用户创建
- 基础权限分配

可以后续再做：

- 表级浏览与执行 SQL
- 数据导入导出
- 备份与恢复
- 性能分析

## 8. 计划任务与扩展运维能力

虽然计划任务目前还没开始，但从技术位置上看，它和站点管理、数据库管理属于同一类“系统控制型能力”。

这类功能有共同特点：

- 需要后端代替前端去操作系统或服务
- 涉及真实环境状态，而不是纯业务数据
- 需要更清晰的错误处理、权限控制和回滚思路

后续如果做计划任务，技术上大概率会围绕：

- `crontab` 读写
- 任务列表展示
- 新增 / 编辑 / 删除任务
- 启停与日志查看

展开。

## 9. 当前技术路线总结

如果把 PanelLab 当前和后续的核心能力抽象一下，可以归纳为三类：

### 9.1 业务型能力

例如：

- 登录注册
- 用户资料
- 配置保存

特点是“数据主要存在数据库里”。

### 9.2 展示型能力

例如：

- 仪表盘
- 概览
- 监控卡片

特点是“主要负责把接口结果组织成用户可读页面”。

### 9.3 系统控制型能力

例如：

- 系统监控
- 远程 SSH 采集
- Nginx 配置管理
- MySQL 管理
- 计划任务管理

特点是“后端要真正接触操作系统或外部服务”。

这也是项目后续最有技术含量、同时风险最高的部分。

---

## 10. 对后续开发的建议

结合当前项目状态，后续技术推进建议按以下顺序：

1. 先收口监控模块和阶段 2 文档，确保已有能力稳定。
2. 再进入阶段 3，先做站点配置 CRUD，再做 Nginx 生效链路。
3. 阶段 4 只做 MySQL 最小闭环，不要一开始铺太大。
4. 在阶段 3 和阶段 4 期间同步补充最小测试与 Linux 环境验证。

这样推进的原因是：

- 当前项目最缺的是“系统控制型能力”的稳定实现
- 阶段 3 和阶段 4 都会进入真实系统操作，复杂度明显高于前两阶段
- 如果不先把边界和技术路线讲清楚，后续很容易在实现时混淆“面板配置”和“系统生效配置”
